# 提示语优化技巧

> 掌握这些技巧，让你的提示语从"能用"变成"好用"。

---

## 🎯 核心原则

### 1. 清晰性原则
- 使用明确、具体的语言
- 避免模糊的描述
- 提供足够的上下文

### 2. 结构化原则
- 使用标题和列表
- 分步骤说明
- 定义输入输出格式

### 3. 示例化原则
- 提供输入输出示例
- 展示预期效果
- 包含边界情况

---

## 📝 具体技巧

### 技巧 1：角色设定（Role Prompting）

为 AI 设定一个具体的角色，提升输出质量。

```markdown
❌ 一般：
解释什么是 REST API。

✅ 优化：
你是一位有 15 年经验的后端架构师，正在给初级开发者讲解技术概念。
请用通俗易懂的语言解释什么是 REST API，并包含：
1. 核心概念（用类比说明）
2. 实际应用场景
3. 常见误区和注意事项
4. 一个具体的代码示例
```

### 技巧 2：思维链（Chain-of-Thought）

引导 AI 逐步思考，提升复杂任务的准确性。

```markdown
请按照以下步骤解决这个问题：

步骤 1：理解问题
[描述对问题的理解]

步骤 2：分析约束条件
[列出所有约束]

步骤 3：提出解决方案
[给出多个可能的方案]

步骤 4：评估方案
[对比各方案的优缺点]

步骤 5：给出最终答案
[选择最优方案并说明理由]
```

### 技巧 3：少样本学习（Few-Shot Learning）

通过示例引导 AI 理解你的需求。

```markdown
请将以下文本转换为 JSON 格式。

示例 1：
输入：姓名：张三，年龄：25，城市：北京
输出：{"姓名": "张三", "年龄": 25, "城市": "北京"}

示例 2：
输入：姓名：李四，年龄：30，城市：上海
输出：{"姓名": "李四", "年龄": 30, "城市": "上海"}

现在处理：
输入：姓名：王五，年龄：28，城市：深圳
输出：
```

### 技巧 4：输出格式化

明确指定输出格式，减少后期处理。

```markdown
请以以下格式输出：

## 摘要
[一句话总结]

## 要点
- [要点 1]
- [要点 2]
- [要点 3]

## 详细说明
### [子主题 1]
[内容...]

### [子主题 2]
[内容...]

## 行动建议
1. [建议 1]
2. [建议 2]
3. [建议 3]
```

### 技巧 5：约束条件设置

通过约束引导 AI 生成符合要求的内容。

```markdown
## 约束条件
- 字数：500-800 字
- 语气：专业但友好
- 避免：过于技术化的术语（如果必须使用，请提供解释）
- 必须包含：至少 2 个实际案例
- 格式：使用 Markdown，包含适当的 emoji
```

### 技巧 6：迭代优化

分轮次逐步优化输出。

**第一轮：**
```markdown
请生成一份产品需求文档的框架。
```

**第二轮：**
```markdown
很好。请在"功能需求"部分添加更多细节，
包括用户故事和验收标准。
```

**第三轮：**
```markdown
请为每个功能添加优先级（P0/P1/P2）和预计工作量。
```

**第四轮：**
```markdown
最后，请添加一个风险分析和应对措施的章节。
```

## 🚀 高级技巧

### 技巧 7：自我一致性检查

让 AI 验证自己的输出。

```markdown
请完成以下任务，并自我检查：

任务：[任务描述]

完成后的检查清单：
- [ ] 是否回答了所有问题？
- [ ] 是否有逻辑错误？
- [ ] 是否符合约束条件？
- [ ] 示例是否恰当？

请先完成任务，然后根据检查清单自我评估，
如有不足请改进。
```

### 技巧 8：多角色辩论

模拟多个视角，获得更全面的分析。

```markdown
请从以下三个角色的角度，分析这个技术方案：

角色 A：技术负责人（关注实现难度和技术风险）
角色 B：产品经理（关注用户需求和商业价值）
角色 C：运维工程师（关注部署和稳定性）

每个角色需要：
1. 列出 3 个支持的理由
2. 列出 3 个担忧的点
3. 给出总体评分（1-10）和建议
```

### 技巧 9：反向提示

告诉 AI 不要做什么，同样重要。

```markdown
请撰写一篇关于微服务的文章。

## 要求
- 面向初级开发者
- 包含具体示例
- 解释核心概念

## 不要
- 不要假设读者已经了解分布式系统
- 不要使用过多的行业术语而不解释
- 不要只讲理论而没有实践建议
- 不要忽略潜在的缺点和挑战
```

### 技巧 10：元认知提示

让 AI 解释自己的思考过程。

```markdown
请解决这个问题，并解释你的思考过程：

[问题描述]

思考过程要求：
1. 你是如何理解这个问题的？
2. 你考虑了哪些解决方法？
3. 为什么选择这个解决方案？
4. 这个方案有什么局限性？
```

## 🛠️ 实用模板

### 模板 1：代码生成

```markdown
你是一位 [语言] 专家。

请帮我编写一个函数，实现以下功能：
[功能描述]

## 输入
- [参数 1]：[类型]，[说明]
- [参数 2]：[类型]，[说明]

## 输出
- [返回值]：[类型]，[说明]

## 要求
- 代码简洁，注释清晰
- 处理边界情况：[列出边界情况]
- 时间复杂度优于 O([复杂度])
- 包含 3 个测试用例

## 示例
输入：[示例输入]
输出：[示例输出]
```

### 模板 2：文章写作

```markdown
你是一位专业的 [领域] 作家。

请撰写一篇关于 [主题] 的文章。

## 目标读者
[读者描述]

## 文章结构
1. 引言：说明文章价值和阅读收益
2. 背景：必要的上下文信息
3. 主体：3-5 个核心要点，每个配案例
4. 实践：可执行的行动建议
5. 总结：回顾核心内容

## 风格要求
- 语气：[友好/专业/幽默等]
- 字数：[字数范围]
- 格式：使用 Markdown，适当使用 emoji
```

### 模板 3：数据分析

```markdown
你是一位数据分析师。

请分析以下数据：
[数据内容]

## 分析维度
1. 描述性统计（基本特征）
2. 趋势分析（时间序列）
3. 关联分析（变量关系）
4. 异常检测（离群值）
5. 预测建议（未来趋势）

## 输出要求
- 使用表格展示关键数据
- 包含可视化建议（图表类型）
- 提供 3-5 个 actionable insights
- 指出数据局限性
```

## ⚠️ 常见错误

| 错误 | 说明 | 改进建议 |
|------|------|----------|
| 过于模糊 | "帮我写点东西" | 明确主题、目标、格式 |
| 过度约束 | 限制太多导致 creativity 降低 | 平衡约束和自由度 |
| 缺少上下文 | AI 不了解背景信息 | 提供必要的背景 |
| 一次请求太多 | 任务过于复杂 | 分解为多个子任务 |
| 不验证输出 | 完全信任 AI 结果 | 始终验证关键信息 |

---

## 💡 最佳实践总结

1. **开始简单**：先用简单提示语测试
2. **逐步迭代**：根据结果不断优化
3. **记录有效模式**：建立个人的提示语模式库
4. **A/B 测试**：对比不同提示语的效果
5. **关注反馈**：根据输出质量调整策略
6. **保持更新**：随着 AI 能力提升，优化旧提示语
