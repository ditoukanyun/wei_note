## 📝 提示语内容

```
你是一位专业的知识管理专家，擅长从大量文本中提取关键信息并进行结构化整理。

请对以下文章进行深度总结：

{{文章内容}}

## 总结要求

### 1. 核心观点（3-5点）
- 提炼文章的核心论点
- 用简洁的语言表达
- 突出最有价值的见解

### 2. 关键要点（5-8点）
- 列出文章中的重要事实和论据
- 使用 bullet points 格式
- 保持逻辑顺序

### 3. 结构梳理
- 文章的整体框架
- 各部分的主要内容
- 论证的逻辑链条

### 4.  actionable insights（可执行的建议）
- 文章提供的实用建议
- 可以立即采取的行动
- 具体的方法和步骤

### 5. 延伸思考
- 文章引发的思考
- 相关的疑问或不同观点
- 值得进一步探索的方向

### 6. 记忆卡片
- 创建3-5个 Q&A 形式的记忆点
- 问题和答案都要简洁明了
- 便于复习和记忆

## 输出格式
请使用 Markdown 格式输出，包含适当的标题层级和列表。

---

附加要求：
- 总结长度：{{长度要求}}（简短/中等/详细）
- 关注重点：{{重点领域}}（如：技术细节、商业逻辑、方法论等）
- 目标受众：{{受众描述}}
```

## 🎯 使用说明

### 适用场景
- 快速阅读长文
- 学习笔记整理
- 文献综述
- 会议记录总结

### 输入变量
| 变量名 | 说明 | 示例 |
|--------|------|------|
| `文章内容` | 需要总结的原文 | 粘贴文章全文 |
| `长度要求` | 总结的详细程度 | 简短/中等/详细 |
| `重点领域` | 希望重点关注的内容 | 技术实现 |
| `受众描述` | 为谁总结 | 技术人员 |

### 参数设置
- **Temperature**: 0.3-0.5
- **Max Tokens**: 根据文章长度调整
- **建议**: 文章过长时先分段处理

## 💡 使用示例

### 示例 1：技术文章总结

**输入：**
```
文章内容：[一篇关于微服务架构的3000字技术文章]
长度要求：中等
重点领域：架构设计原则
目标受众：后端开发工程师
```

**输出：**
```markdown
## 核心观点

1. **微服务的本质**是围绕业务能力构建的分布式系统，而非简单的技术拆分
2. **数据独立性**是微服务成功的关键，避免数据库共享导致的耦合
3. **演进式设计**比一步到位更重要，应该从单体逐步拆分

## 关键要点

- 微服务适用于复杂度高的系统，简单系统使用单体更合适
- 每个服务应该有独立的代码库、数据库和部署管道
- API 网关是统一入口，处理认证、限流、路由等横切关注点
- 分布式事务应该尽量避免，采用最终一致性设计
- 监控和日志聚合是运维微服务的必备能力

## 可执行建议

1. **评估当前系统复杂度**，确定是否真的需要微服务
2. **识别业务边界**，按照领域驱动设计划分服务
3. **建立 DevOps 文化**，自动化是微服务成功的基础
4. **先拆分数据**，再拆分服务，降低风险

[其他部分...]
```
